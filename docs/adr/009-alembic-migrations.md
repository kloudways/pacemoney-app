---
title: Use Alembic for versioned DB migrations instead of create_all
date: 2026-07-20
status: accepted
---

## Context

The application originally called `Base.metadata.create_all()` inside the FastAPI lifespan event. This is sufficient to bootstrap an empty database but cannot evolve an existing schema — adding or removing a column requires manual intervention or a full table drop.

## Decision

Replace `create_all()` in the lifespan with `alembic_command.upgrade(cfg, "head")`. Alembic tracks applied migrations in the `alembic_version` table and runs only the migrations that have not yet been applied.

Tests continue to use `create_all()` and `drop_all()` directly via the `conftest.py` fixture. Alembic is not involved in the test path — this keeps tests fast and avoids requiring a migration file for every test schema change.

## Consequences

- Migrations must be written (or autogenerated with `alembic revision --autogenerate`) for every schema change.
- A column present in the ORM model but absent from the migrations causes a `ProgrammingError` at runtime — this happened in Phase 9 when `created_at` was omitted from the initial migration.
- `alembic.ini` must include full logging config sections (`[loggers]`, `[handlers]`, `[formatters]`) for `fileConfig()` to succeed — the minimal ini is not sufficient.
- The `alembic/` directory and `alembic.ini` must be present in the Docker image; they are copied explicitly in the Dockerfile before `app/`.
