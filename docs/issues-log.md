# Issues log

All application and pipeline issues encountered during Phases 1-6 of the Pace Money build.

| Issue | Phase | Symptom | Root cause | Fix | Prevention |
|-------|-------|---------|-----------|-----|-----------|
| `python3-venv` missing on Jenkins | 5 | `python3 -m venv .venv` failed during Unit Tests stage | Ansible common role did not install `python3-venv` or `python3-pip` | Added both packages to the Ansible common role and installed live via SSH | Include the full Python toolchain in the base Ansible role |
| SonarQube missing `sonar.organization` | 5 | Scanner exited with "You must define the following mandatory properties: sonar.organization" | `sonar-project.properties` did not include the organisation key | Added `sonar.organization=kloudways` to `sonar-project.properties` | Include the organisation key when setting up `sonar-project.properties` |
| SonarQube wrong project key | 5 | Scanner could not find the project in SonarCloud | Project key in `sonar-project.properties` was set to `pacemoney` instead of the correct `kloudways_pacemoney-app` | Corrected the key to match the value shown in the SonarCloud project URL | Copy the project key from the SonarCloud UI when configuring the properties file |
| SonarCloud Automatic Analysis conflict | 5 | Scanner reported "Automatic Analysis is enabled, you cannot run CI-based analysis" | The SonarCloud project had Automatic Analysis enabled, which conflicts with CI-based scanning | Disabled Automatic Analysis in SonarCloud project settings under Administration -> Analysis Method | Disable Automatic Analysis when creating the project if CI-based scanning is intended |
| Checkov CKV_DOCKER_2 (missing HEALTHCHECK) | 5 | Checkov reported one failed Dockerfile check | Dockerfile had no HEALTHCHECK instruction | Added a HEALTHCHECK using Python urllib to verify the `/health` endpoint | Include HEALTHCHECK in the Dockerfile from the start |
| `requirements.txt not found` in Docker build | 5 | `pip install -r requirements-prod.txt` failed because `requirements.txt` was missing from the build context | Dockerfile only copied `requirements-prod.txt`; `requirements-prod.txt` references `-r requirements.txt` transitively | Changed the COPY instruction to `COPY requirements.txt requirements-prod.txt ./` | Copy all files that are transitively referenced by pip install |
| Trivy blocks build on unfixed CVEs | 5 | Trivy exited with code 1; 22 HIGH/CRITICAL CVEs found in Debian base image | Base OS CVEs all had status "affected" with no fixed version available | Added `--ignore-unfixed` to the Trivy invocation | Use `--ignore-unfixed` from the start; accept that base OS CVEs with no upstream fix are not actionable |
| `db-url` credential contained placeholder value | 5 | Helm deploy succeeded but pods crashed with SQLite errors; `DATABASE_URL` secret contained `<your_db_password>` literally | The Jenkins credential was saved with the template placeholder instead of the real RDS password | Updated the Jenkins credential with the actual RDS password retrieved from Terraform state | Verify credential values in Jenkins before triggering builds; never copy placeholder values from documentation |
| Pods crash silently (exit code 137, empty logs) | 5 | Pods in CrashLoopBackOff; `kubectl logs --previous` returned nothing; liveness probe reported `connection refused` | Two bugs combined: (1) `Base.metadata.create_all` at module level blocked the process on a TCP connection to RDS before uvicorn bound to port 8000; (2) the RDS security group did not allow inbound traffic from kops nodes. The liveness probe killed the hanging process after 40 seconds, before any log output was written. | Moved `create_all` to a FastAPI lifespan event; added a second RDS security group ingress rule allowing port 5432 from the VPC CIDR (10.1.0.0/16) | Never connect to external services at module import time; ensure security groups cover all compute that needs access |
| Unit tests fail after lifespan refactor | 5 | `sqlalchemy.exc.OperationalError: no such table: transactions` for two tests | `TestClient(app)` at module level does not trigger the FastAPI lifespan, so `create_all` never ran during tests | Added `tests/conftest.py` with a session-scoped autouse fixture that calls `create_all` before tests and `drop_all` after | When using lifespan for database setup, ensure the test suite has an equivalent fixture |
| ZAP container exits immediately with usage message | 5 | ZAP printed its help text and exited without scanning | `-r zap-report.html` specifies a file path, but `/zap/wrk` was not mounted so ZAP refused to proceed | Added `-v ${WORKSPACE}:/zap/wrk:rw` to the docker run command | Always mount `/zap/wrk` when using ZAP file-based options |
| ZAP report write fails with `AccessDeniedException` | 5 | ZAP completed the scan but could not write `zap-report.html`; scan results appear in console only | The ZAP container runs as a UID that does not match the Jenkins workspace file owner | Not fixed; scan results are available in the console log. Build is not affected due to `|| true`. | To fix properly, pre-create the report file with open permissions or match the ZAP container UID to the Jenkins workspace UID |
| Manual prometheus metrics did not cover all routes | 6 | `/health` and `DELETE /transactions` had no metrics | The original implementation used a manually incremented `Counter` only wired to two routes | Replaced with `prometheus-fastapi-instrumentator` which auto-instruments all routes with histograms | Use instrumentation libraries rather than manual metric tracking |
| Test DB had stale data between runs | 6 | `test_list_transactions_empty_on_fresh_db` failed with unexpected data | The conftest fixture called `create_all` but not `drop_all` at the start, so data from a previous run persisted in the SQLite file | Changed conftest to call `drop_all` before `create_all` at session start | Always drop and recreate tables at the start of the test session to guarantee a clean state |
| Community Grafana dashboards require Loki | 6 | Dashboard import blocked by missing Loki data source | Dashboards 16110 and 17175 both depend on Loki for log panels; Loki is not installed | Built a manual dashboard in Grafana with two panels: Request Rate and P95 Latency | Use Prometheus-only dashboards or install Loki if log panels are needed |
